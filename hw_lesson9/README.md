Чотири принципи ООП

Інкапсуляція: приховування внутрішнього стану об’єкта; доступ через методи/інтерфейси (control visibility).

Абстракція: виділення суттєвих властивостей і приховування деталей реалізації; працювати через контракт (інтерфейс/абстрактний клас).

Наслідування: повторне використання коду і побудова ієрархій (is‑a), коли підкласи розширюють поведінку базового класу.

Поліморфізм: можливість використовувати об’єкти різних класів через єдиний інтерфейс; різні реалізації викликаються однаково (динамічне/статичне зв’язування).


SOLID — це набір принципів для об’єктно-орієнтованого дизайну, який допомагає робити код гнучким, зрозумілим і легко підтримуваним.

S — Single Responsibility Principle (Принцип єдиної відповідальності)

Клас/модуль має мати тільки одну причину для зміни.
Якщо він робить більше одного, його краще розбити.
Наприклад: клас Order не має одночасно обробляти логіку замовлення і друкувати чек.
O — Open/Closed Principle (Принцип відкритості/закритості)

Система має бути відкритою для розширення, але закритою для модифікації.
Нову поведінку додаємо через нові класи або реалізації, а не змінюємо вже готовий код.
Це знижує ризик зламу існуючих функцій при додаванні нового.
L — Liskov Substitution Principle (Принцип підстановки Лісков)

Підкласи повинні повністю замінювати базовий клас без зміни коректності програми.
Якщо B наслідує A, то об’єкти B мають працювати там, де очікується A.
Інакше ієрархія успадкування порушується.
I — Interface Segregation Principle (Принцип розділення інтерфейсів)

Клієнти не повинні залежати від методів, які вони не використовують.
Краще мати багато вузьких інтерфейсів, ніж один великий “жирний”.
Це робить код менш зв’язаним і легшим для тестування.
D — Dependency Inversion Principle (Принцип інверсії залежностей)

Високорівневі модулі не повинні залежати від низькорівневих; обидва повинні залежати від абстракцій.
Абстракції не повинні залежати від деталей; деталі повинні залежати від абстракцій.
Це дозволяє замінювати реалізації без зміни бізнес-логіки.


DRY: не дублюй логіку; кожна інформація/поведінка має єдине джерело істини. Вирішення: винести повторюваний код у функцію/модуль, інтерфейс або утиліту; використовувати композицію або спадкування там, де це природно.


KISS: роби просто; рішення повинні бути зрозумілими і мінімальними. Уникай передчасної абстракції і надскладної архітектури — простіші реалізації легше тестувати й підтримувати.
